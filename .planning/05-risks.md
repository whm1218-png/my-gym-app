# 위험 요소 및 대응 방안

프로젝트 전제: Flutter, 로컬 DB(SQLite/drift 권장), 다크 모드 중심 UX, Epley 기반 e1RM, 12주 중간 발표/15주 최종 발표.

## 위험 식별 (5가지)

### 1) 기술 위험: 로컬 DB 선택/구현 지연
- **설명:** SQLite(drfit)와 Hive 사이 결정이 늦어지면 데이터 계층 구현이 지연되고 화면 개발이 막힌다.
- **영향:** 일정 지연, 반복 리팩터링.
- **대응 방안:**
  - 2주차 내 간단 POC(세션 생성/조회 1개 시나리오)로 저장소를 확정한다.
  - 확정 후 스키마 변경은 마이그레이션 규칙으로만 처리하고 임의 변경을 제한한다.

### 2) 기술 위험: 과부하/1RM 계산 로직 불일치
- **설명:** 직전 세션 비교 기준, 대표 세트 선정 기준이 모호하면 결과 신뢰가 떨어진다.
- **영향:** 사용자 혼란, 기능 신뢰도 하락.
- **대응 방안:**
  - 계산 규칙을 단일 모듈로 고정하고 문서(`04-data-model.md`)와 코드 주석을 동기화한다.
  - 단위 테스트 케이스(중량 증가/반복 증가/동일/감소)를 최소 세트로 운영한다.

### 3) 일정 위험: 발표 직전 기능 확장 과다
- **설명:** 중간 발표 직전 새로운 Should 기능을 무리하게 넣으면 핵심 데모 안정성이 떨어진다.
- **영향:** 크래시·데모 실패 가능성 증가.
- **대응 방안:**
  - 11주차에 feature freeze를 걸고 12주차는 버그 수정/데모 리허설에 집중한다.
  - Must 미완료 상태에서는 Should/Could 개발을 제한한다.

### 4) 운영 위험: 로컬 저장 한계로 인한 데이터 소실
- **설명:** 서버 동기화가 없으므로 앱 삭제/기기 분실 시 데이터 손실 가능성이 존재한다.
- **영향:** 사용자 신뢰 저하, 평가 질의 대응 어려움.
- **대응 방안:**
  - 온보딩/설정에 로컬 저장 한계를 명확히 안내한다.
  - 시간이 허용되면 Could로 JSON/CSV 내보내기 기능을 추가해 최소 백업 수단을 제공한다.

### 5) AI 의존 위험: 생성 결과 오용/검증 누락
- **설명:** AI가 작성한 코드/문서를 검증 없이 채택하면 잘못된 로직 또는 부정확한 문서가 남을 수 있다.
- **영향:** 버그 누적, 발표 질의 대응 실패, 팀 학습 저하.
- **대응 방안:**
  - AI 산출물은 항상 팀원이 리뷰하고 “근거 문서/테스트/실행 확인” 3종 체크 후 반영한다.
  - 계산식, 일정, 범위 의사결정은 회의록과 ADR에 남겨 추적성을 확보한다.

## 위험 관리 운영 원칙
- 주간 점검: 매주 위험 항목의 상태(신규/완화/종결)를 10분 내 점검.
- 변경 관리: 범위 변경은 WBS와 ADR에 반영 후 시행.
- 발표 보호: 12주/15주 전 주간은 안정화 우선.
