# 핵심 요구사항: 사용자 시나리오 + MoSCoW

## 핵심 사용자 시나리오 (3개)

### 시나리오 1: 운동 중 빠른 기록 + 과부하 피드백
- **입문·중급** 사용자는 운동 세션에서 각 세트의 **중량·반복**(및 필요 시 완료 여부)을 최소 탭으로 기록한다.
- 앱은 **같은 운동**에 대해 **직전 종료 세션**과 비교해 중량 또는 반복이 늘었는지 등을 즉시 보여준다.
- 사용자는 세션 중 “**오늘 과부하 달성 여부**”를 배지 또는 한 줄 설명으로 확인한다.
- 입문자를 위해 **용어 설명**(예: “과부하 = 지난번보다 무겁게 또는 더 많이”)을 첫 실행 또는 도움말에서 제공할 수 있다.

### 시나리오 2: 세션 후 예상 1RM(e1RM) 확인
- 사용자가 벤치프레스·스쿼트·데드리프트 등을 기록하면, 앱은 **Epley 공식**으로 세트별 e1RM을 구하고 **운동당 대표값**(정책은 `04-data-model.md`와 동일)을 보여준다.
- **Should** 범위에서 주·월 단위 **추세 그래프**로 전주·전월 대비 변화를 본다(그래프는 MVP 이후 단계에서도 우선 구현 권장).
- 사용자는 숫자와 추세를 바탕으로 다음 세션 목표를 정한다(자동 추천은 Should).

### 시나리오 3: 기기 안에서 기록의 연속성
- 계정 없이 **기기 내부 DB**에 모든 세션이 저장된다.
- 사용자는 **기록 목록·상세**에서 과거 세션, 과부하 이력, e1RM 변화를 **동일 기기**에서 끊김 없이 조회한다.
- 앱 삭제·기기 분실 시 데이터는 소실될 수 있음을 **설정 또는 온보딩**에서 한 줄로 안내한다(향후 내보내기로 보완).

## 기능 분류 (MoSCoW)

### Must (필수)
- 운동/세트 기록: 운동명, 중량, 반복, 세트, 세션 일시 (**로컬 저장**)
- 점진적 과부하 판정: 직전 세션 대비 **중량·반복** 증가 기준 (`04-data-model.md` 규칙)
- **Epley** 기반 예상 1RM 계산 및 운동별 표시
- 운동별 최근 기록·히스토리 조회
- 기본 대시보드: 오늘·최근 세션 요약, 최근 성장 지표
- **다크 모드** 기본 또는 강조 테마(시인성·큰 숫자·터치 영역)

### Should (중요)
- 운동별 1RM(e1RM) 추세 그래프(주/월 단위)
- 다음 세션 목표 추천(예: +2.5kg 또는 +1 rep 규칙 기반)
- 개인 기록(PR) 자동 하이라이트(e1RM 역대 최고 갱신 시)
- 세션 템플릿(자주 하는 루틴 저장/재사용)
- 입문자용 **온보딩/툴팁**(과부하·e1RM 의미)

### Could (있으면 좋음)
- JSON/CSV **내보내기**(백업·보고서용; 서버 없이 파일로 저장)
- 운동 휴식 타이머 및 로컬 알림
- 부위별 볼륨 통계
- 라이트 모드 토글(다크가 기본이어도 선택권)

### Won’t (초기 단계·본 과제 범위에서 제외)
- **서버·계정·클라우드 동기화**(초기 복잡도 축소; 이후 확장 과제로 분리 가능)
- 코치–회원 B2B, 실시간 소셜 피드
- 생체 센서(HRV 등) 기반 자동 최적화
- AI 영상 폼 분석

## 요구사항 전제 (확정)
| 항목 | 내용 |
|------|------|
| 타깃 | **입문자 + 중급자** (기록·성장 집중) |
| 플랫폼 | **Flutter** |
| 데이터 | **로컬만** (SQLite 또는 Hive) |

## 저장소 선택 가이드 (SQLite vs Hive)
- **SQLite + drift(또는 sqflite)**: 관계형 쿼리(직전 세션, 운동별 히스토리)에 유리. 본 앱 데이터 모델과 궁합이 좋다.
- **Hive**: 스키마가 단순하고 NoSQL스런 맵 구조로 충분할 때 빠른 개발에 유리. 복잡한 조인·리포트 쿼리는 앱 코드에서 집계.
- **권장**: 관계·조회가 많으므로 **SQLite 계열(drift 권장)**을 기본안으로 두고, 팀 역량·일정에 따라 Hive로 단순화 가능함을 문서에만 명시.
