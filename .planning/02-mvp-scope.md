# 개발 범위 및 학기 마일스톤

`00-vision.md`, `01-requirements.md`와 일치하게 **Flutter · 로컬 저장 · 다크 모드 · Epley · 입문·중급 타깃**을 전제한다. 세부 일정·작업 분해는 **`05-wbs-schedule.md`**를 따른다.

## 산출물 정의 (최종 발표 시)

**“기기에서 로그인 없이 세션 단위 운동·세트를 기록하고, 직전 세션 대비 점진적 과부하와 Epley e1RM을 운동별로 확인하며, 다크 테마 기반 UI로 헬스장에서 사용 가능한 하나의 출시 가능한 형태의 앱.”**

- 서버·클라우드 동기화는 포함하지 않는다(Won’t).
- 로컬 DB는 **SQLite(drift 등) 또는 Hive** 중 택 일(`01-requirements.md` 가이드).

## 단계별 범위 (기능 관점)

### Phase A — 필수 기능 (Must 동결)
- Flutter 프로젝트, 다크 테마, 기본 라우팅·탭(홈/기록/설정).
- 로컬 영속 레이어: 세션 · 세션 내 운동 · 세트 CRUD.
- 직전 세션 대비 과부하 판정, Epley e1RM, 세션 요약·대시보드 최소 버전.

### Phase B — 데모·중간발표 준비(12주 차)
- Must 전부 완료 + 화면 전환 안정화, 크래시 최소화, **중간발표 데모 시나리오**(3~5분) 리허설 가능 수준.
- Should 중 **일부**(예: e1RM 추세 그래프 또는 온보딩)를 선택해 포함하면 평가에 유리(일정 허용 시).

### Phase C — 완성·최종발표 준비(13~15주)
- 나머지 Should 항목·Could(내보내기 등) 선택 구현.
- 문서화(기능 설명·스크린샷)·발표 자료·**최종 데모** 동선 확정.

## 중간발표(12주)·최종발표(15주)에 맞춘 기대 수준

| 시점 | 기대 수준 |
|------|-----------|
| **12주 (중간)** | Must 충족 데모. “운동 기록 → 과부하·e1RM 확인” 플로우가 끊기지 않음. 다크 UI 가시성 확보. |
| **15주 (최종)** | Must + 우선순위 Should 반영. 사용 시나리오·한계(로컬만)·향후 과제(서버) 명시 가능. |

## 리스크

- **SQLite vs Hive 선택 지연**: 1~2주 내 POC로 CRUD 경로 하나를 정하고 고정한다.
- **데모 일정**: 12주 주간에 **코드 동결**(feature freeze)·**데모 플래그** 원칙을 둔다.
- **데이터 소실 안내**: 앱 내 문구로 사용자 기대치를 관리하고, 가능하면 Phase C에서 Could 내보내기를 검토한다.
