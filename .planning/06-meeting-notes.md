# 회의록

## 회의 개요
- **일시:** 2026-05-11
- **주제:** 헬스 기록 앱 기술 스택 및 기획 확정
- **목표:** 프로젝트 핵심 의사결정 확정 및 실행 문서 정리

## 논의 내용 요약
- 타깃 사용자를 운동 기록과 성장에 집중하는 입문자·중급자로 재정의했다.
- 플랫폼은 Flutter로 확정하고, 일정 내 구현 안정성 및 다크 모드 적용 용이성을 주요 근거로 삼았다.
- 데이터 저장은 초기 단계에서 서버 연동 없이 로컬 DB 중심으로 진행하기로 합의했다.
- 1RM 계산은 Epley 공식을 기본으로 적용하고, 과부하 판단은 중량/반복 증가 기준을 유지한다.
- 일정은 12주 중간 발표, 15주 최종 발표를 기준으로 WBS를 구성하고 발표 전 안정화 구간을 확보하기로 했다.

## 최종 결정 사항
1. **모바일 프레임워크:** Flutter
2. **데이터 저장 방식:** 로컬 DB (SQLite/drift 권장, Hive 대안)
3. **1RM 계산 공식:** Epley
4. **UI/UX:** 다크 모드 중심, 헬스장 시인성 우선
5. **범위 관리:** Must 우선, Should/Could는 중간 발표 안정화 이후 단계적 반영

## 산출물 업데이트
- `00-vision.md`: 비전/목표 및 기술 스택 정합성 반영
- `01-requirements.md`: 시나리오, MoSCoW, 저장소 가이드 반영
- `02-mvp-scope.md` ~ `05-wbs-schedule.md`: 일정/실행 계획 반영
- `decisions/ADR-0001~0003`: 핵심 의사결정 로그 작성
- `05-risks.md`: 위험요소 및 대응 전략 정리

## 후속 액션 아이템
- 저장소 POC(2주차 내): SQLite/drift 기반 세션 생성·조회 시나리오 검증
- 도메인 테스트: Epley/과부하 규칙 단위 테스트 우선 작성
- 데모 관리: 11주차 기능 동결, 12주차 발표 리허설 집중

4. **12주 중간 발표 데모에서 반드시 보여줘야 할 핵심 사용자 여정 1~2개는 무엇이며, 실패 허용 범위는 어디까지인가?**
5. **AI 사용 시 팀의 검증 기준(코드 리뷰, 테스트 통과, 문서 근거)은 어디까지를 ‘반드시’로 둘 것인가?**
