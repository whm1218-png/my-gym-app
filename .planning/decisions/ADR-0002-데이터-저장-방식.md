# ADR-0002: 데이터 저장 방식 (로컬 DB)

## 상태
채택됨 (Accepted)

## 배경
- 현재 단계는 서버/계정/클라우드 동기화를 제외하고 로컬 중심 MVP를 우선 완성하는 것이 목표다.
- 핵심 기능은 세션/운동/세트 기록, 직전 세션 비교, e1RM 계산으로, 데이터의 조회 패턴이 비교적 구조적이다.
- 발표 일정(12주/15주)을 고려하면 백엔드 구축보다 앱 핵심 가치 검증에 집중해야 한다.

## 대안
1. SQLite 계열 (drift 또는 sqflite)
2. Hive
3. Firebase/서버 연동 기반 원격 저장

## 결정
- 저장 방식은 로컬 DB로 고정한다.
- 기본 권장안은 SQLite 계열(Flutter에서는 drift 우선)로 한다.
- 단, 팀 역량/일정 리스크가 클 경우 Hive로 단순화할 수 있도록 문서상 대안을 유지한다.

## 이유
- 직전 세션 조회, 운동별 히스토리, 정렬/필터링 등 관계형 조회가 많아 SQLite가 유리하다.
- drift는 타입 안전 쿼리와 마이그레이션 관리가 상대적으로 명확해 중장기 유지보수에 도움이 된다.
- 서버를 배제하면 네트워크/인증/동기화 리스크를 줄이고 핵심 UX와 도메인 로직 완성도에 집중할 수 있다.

## 결과/영향
- 초기 릴리스는 단일 기기 데이터 지속성만 보장하며, 앱 삭제/기기 변경 시 데이터 소실 가능성을 안내한다.
- 추후 확장 시 내보내기(JSON/CSV) 또는 서버 동기화 단계를 별도 ADR로 재논의한다.
- 데이터 모델은 로컬 단일 사용자 전제에 맞춰 설계한다.
